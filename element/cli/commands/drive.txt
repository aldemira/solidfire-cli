                     <entry colname="col1" cid="1Thqa5">
                        <apiname>SecureEraseDrives</apiname> enables you to remove any residual data from <parmname>drives</parmname that have a status of "available." You might want to use this method when replacing a drive nearing the end of its service life that contained sensitive data. This method uses a Security Erase Unit command to write a predetermined pattern to the drive and resets the encryption key on the drive. This asynchronous method might take up to two minutes to complete. You can use GetAsyncResult to check on the status of the secure erase operation. You can use the ListDrives method to obtain the driveIDs for the <parmname>drives</parmname you want to secure erase. 
                     </entry>
                     <entry colname="col2" cid="sKwWv">
                        <parmname>drives</parmname>
                        : List of driveIDs to be secure erased.

                  </row>

               <tbody cid="7uqUX">
                  <row cid="2EBjiH">
                     <entry colname="col1" cid="1Thqa5">
                        You can use <apiname>RemoveDrives</apiname> to proactively remove <parmname>drives</parmname that are part of the cluster. You might want to use this method when reducing cluster capacity or preparing to replace <parmname>drives</parmname nearing the end of their service life. Any data on the <parmname>drives</parmname is removed and migrated to other <parmname>drives</parmname in the cluster before the drive is removed from the cluster. This is an asynchronous method. Depending on the total capacity of the <parmname>drives</parmname being removed, it might take several minutes to migrate all of the data. Use the GetAsyncResult method to check the status of the remove operation. When removing multiple <parmname>drives</parmname, use a single <apiname>RemoveDrives</apiname> method call rather than multiple individual methods with a single drive each. This reduces the amount of data balancing that must occur to even stabilize the storage load on the cluster. You can also remove <parmname>drives</parmname with a "failed" status using <apiname>RemoveDrives</apiname>. When you remove a drive with a "failed" status it is not returned to an "available" or active status. The drive is unavailable for use in the cluster. Use the ListDrives method to obtain the driveIDs for the <parmname>drives</parmname you want to remove. 
                     </entry>
                     <entry colname="col2" cid="sKwWv">
                        <parmname>drives</parmname>
                        : List of driveIDs to remove from the cluster. 
                        <parmname>forceduringupgrade</parmname>
                        : If you want to remove a drive during upgrade, this must be set to true. 
                     </entry>
                  </row>
               </tbody>

                  </row>

               <tbody cid="7uqUX">
                  <row cid="2EBjiH">
                     <entry colname="col1" cid="1Thqa5">
                        <apiname>AddDrives</apiname> enables you to add one or more available <parmname>drives</parmname to the cluster, enabling the <parmname>drives</parmname to host a portion of the cluster's data. When you add a node to the cluster or install new <parmname>drives</parmname in an existing node, the new <parmname>drives</parmname are marked as "available" and must be added via <apiname>AddDrives</apiname> before they can be utilized. Use the ListDrives method to display <parmname>drives</parmname that are "available" to be added. When you add multiple <parmname>drives</parmname, it is more efficient to add them in a single <apiname>AddDrives</apiname> method call rather than multiple individual methods with a single drive each. This reduces the amount of data balancing that must occur to stabilize the storage load on the cluster. When you add a drive, the system automatically determines the "type" of drive it should be. The method is asynchronous and returns immediately. However, it can take some time for the data in the cluster to be rebalanced using the newly added <parmname>drives</parmname. As the new <parmname>drives</parmname are syncing on the system, you can use the ListSyncJobs method to see how the <parmname>drives</parmname are being rebalanced and the progress of adding the new drive. You can also use the GetAsyncResult method to query the method's returned asyncHandle. 
                     </entry>
                     <entry colname="col2" cid="sKwWv">
                        <parmname>drives</parmname>
                        : Returns information about each drive to be added to the cluster. Possible values are: driveID: The ID of the drive to add. (Integer) type: (Optional) The type of drive to add. Valid values are "slice" or "block". If omitted, the system assigns the correct type. (String) 
                        <parmname>forceduringupgrade</parmname>
                        : Allows the user to force the addition of <parmname>drives</parmname during an upgrade. 
                     </entry>
                  </row>
               </tbody>
            </tgroup>
         </table>
